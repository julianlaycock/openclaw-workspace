## Bugs Found & Fixed

### 1. ❌ CRITICAL: Mojibake in security.ts
**File:** `src/backend/middleware/security.ts`  
**Issue:** Orphan UTF-8 byte `0x97` and literal `\r` characters (backslash-r as text) instead of proper CRLF line endings  
**Error:** `TS1127: Invalid character` at line 29, col 42  
**Root Cause:** Previous mojibake-fixing script corrupted the file  
**Fix Applied:**
- Replaced orphan `0x97` with em-dash (`—`)
- Converted literal `\r\n` text to proper CRLF (`0d 0a`)

**Status:** ✅ FIXED

---

### 2. ❌ CRITICAL: Duplicate function in api-key-service.ts
**File:** `src/backend/services/api-key-service.ts`  
**Issue:** `rotateApiKey` function defined twice (lines 199-228 and 232-261)  
**Error:** `Multiple exports with the same name "rotateApiKey"`  
**Root Cause:** Copy-paste error or merge conflict  
**Fix Applied:** Removed duplicate function (lines 232-261)

**Status:** ✅ FIXED

---

### 3. ❌ CRITICAL: event-repository.ts returns void instead of Promise
**File:** `src/backend/repositories/event-repository.ts`, line 70  
**Issue:** `dispatchEvent()` doesn't return a Promise, causing `.catch()` to fail  
**Error:** `Cannot read properties of undefined (reading 'catch')`  
**Impact:** Due diligence endpoint crashes when trying to create audit events  
**Root Cause:** `dispatchEvent` calls `broadcastEvent().catch()` but doesn't return it  
**Fix Applied:**
- Changed `webhook-service.ts` line 121:  
  `broadcastEvent(eventType, payload).catch(() => {});` →  
  `return broadcastEvent(eventType, payload);`

**Status:** ✅ FIXED

---

### 4. ❌ CRITICAL: SQL placeholders wrong in event-repository.ts
**File:** `src/backend/repositories/event-repository.ts`, line 43  
**Issue:** INSERT uses SQLite-style `?` placeholders instead of PostgreSQL `$1, $2, ...`  
**Error:** Would fail silently or cause SQL syntax errors  
**Fix Applied:** Changed placeholders to `$1, $2, $3, $4, $5, $6, $7`

**Status:** ✅ FIXED

---

### 5. ❌ HIGH: event-repository hardcodes DEFAULT_TENANT_ID
**File:** `src/backend/repositories/event-repository.ts`, line 46  
**Issue:** Always uses `DEFAULT_TENANT_ID` instead of `input.tenant_id`  
**Impact:** Audit events not properly isolated by tenant  
**Fix Applied:** Changed to `input.tenant_id || DEFAULT_TENANT_ID`

**Status:** ✅ FIXED

---

### 6. ❌ CRITICAL: queryWithTenant argument order reversed in filing-workflow-service.ts

**Issue:** Calls `queryWithTenant(tenantId, sql, params)` but function signature is `queryWithTenant(sql, params, tenantId)`

## Code Quality Issues

### 1. Inconsistent API Signatures
`queryWithTenant(sql, params, tenantId)` vs. sub-agent usage `queryWithTenant(tenantId, sql, params)`  
→ **Recommendation:** Add TypeScript overload or rename to make parameter order obvious

### 2. Lack of Input Validation
Filing routes don't validate:
- `reportingPeriod` format (should match `YYYY-QN`)
- `nca` enum (should be one of: bafin, cssf, ...)
- `fundId` existence before INSERT

→ **Recommendation:** Add Zod schemas or joi validation

### 3. No Transaction Rollback on Partial Failures
`createFiling` inserts into `filing_submissions` and `filing_status_history` separately.  
If second INSERT fails, first one is orphaned.

→ **Recommendation:** Wrap in explicit transaction in filing-workflow-service
