Versioning Strategy

  • URL-based versioning: /api/v1/, /api/v2/, etc.
  • Major version increments only for breaking changes
  • Multiple versions may be supported simultaneously during transition periods
  • Public API endpoints (/api/v1/public/) follow the same versioning

Deprecation Procedure

Minimum notice period: 6 months

1. Announce Deprecation→ 2. Add Deprecation Header→ 3. Update Documentation→ 4. Migration Guide→ 5. Sunset (6+ months)

Step 1: Announce

  • Notify API consumers via email and changelog
  • Publish deprecation notice in API docs

Step 2: Deprecation Header

  • Add Deprecation: true header to deprecated endpoints
  • Add Sunset: [date] header with planned removal date
  • Add Link: <migration-guide-url>; rel="deprecation" header

Step 3: Documentation

  • Mark deprecated endpoints in OpenAPI spec with deprecated: true
  • Document replacement endpoints

Step 4: Migration Guide

  • Provide endpoint mapping (old → new)
  • Code examples for migration
  • Breaking change details

Step 5: Sunset

  • Remove deprecated endpoints after sunset date
  • Return 410 Gone for removed endpoints with migration pointer
