Scenario 1: Railway Platform Outage

⚠️ Current mitigation: None. Single-region Railway deployment = single point of failure.

Impact

  • Frontend, API, and database all become unavailable
  • No failover or redundancy exists
  • Estimated time to resolution: depends entirely on Railway (typically 15-60 min)

Immediate Actions

  1. Check Railway Status Page
  2. Notify affected customers (see Incident Response Playbook)
  3. Monitor Railway status for recovery

Recommendations

  • Add Cloudflare DNS with failover page for graceful degradation
  • Consider multi-region deployment or standby on Fly.io/Render
  • Implement external uptime monitoring (UptimeRobot, Betterstack)

Scenario 2: Database Loss / Corruption

Backup Procedure

✅ Existing: scripts/backup-db.ts and scripts/backup-db.ps1
  1. Run backup: npx tsx scripts/backup-db.ts
  2. Backup creates a pg_dump file with full schema + data
  3. Store backups off-Railway (S3, local machine, etc.)

Restore Procedure

  1. Create new Railway PostgreSQL instance
  2. Connect via psql using Railway connection string
  3. Restore: psql $DATABASE_URL < backup_file.sql
  4. Update DATABASE_URL env var in Railway service
  5. Redeploy API server
  6. Verify via health check: curl https://www.caelith.tech/api/health/ready

Point-in-Time Recovery

⚠️ Railway PostgreSQL does not offer built-in PITR. For PITR capability, migrate to a managed provider (Supabase, Neon, or AWS RDS) that supports WAL archiving.

Scenario 3: DNS / SSL Issues

Symptoms

  • Site unreachable but Railway services are healthy
  • SSL certificate errors in browser
  • DNS resolution failures

Actions

  1. Check DNS propagation: dig caelith.tech or use dnschecker.org
  2. If using Cloudflare: check Cloudflare dashboard for certificate status
  3. Railway custom domains: verify CNAME records point to Railway
  4. SSL renewal: Railway auto-renews via Let's Encrypt; if failed, remove and re-add custom domain

Recommendation

  • Move DNS to Cloudflare for DDoS protection, caching, and faster propagation
  • Enable "Always Use HTTPS" and set minimum TLS to 1.2

Scenario 4: Security Breach

  1. Isolate: Disable Railway deployments, rotate all secrets
  2. Assess: Check audit_trail table for unauthorized access patterns
  3. Rotate: JWT_SECRET, database password, API keys, session cookies
  4. Notify: If PII compromised → GDPR notification within 72 hours
  5. Remediate: Patch vulnerability, redeploy, monitor
