Escalation Path

  1. 0-5 min: Automated health check detects issue → Discord #dev alert
  2. 5-15 min: Julian notified via Discord/push notification
  3. 15-60 min: If P1 unresolved → consider Railway support ticket
  4. 60+ min: Customer communication if service remains down

Runbook: Site Down

  1. Check Railway dashboard for service status
  2. Check https://status.railway.app
  3. Verify DNS: dig www.caelith.tech
  4. Check health endpoint: curl -I https://www.caelith.tech/api/health/ready
  5. If Railway is up but service crashed → check logs: Railway dashboard → Deployments → Logs
  6. If OOM or crash loop → restart service in Railway
  7. If deploy broke it → rollback to previous deployment in Railway

Runbook: Database Connection Errors

  1. Check Railway PostgreSQL service status
  2. Verify DATABASE_URL env var is correct
  3. Check connection pool exhaustion in logs (look for "too many connections")
  4. If pool exhausted → restart API service to release connections
  5. If database unreachable → Railway PostgreSQL may need restart
  6. If data corruption suspected → see Disaster Recovery Plan

Runbook: Rate Limit Storm

  1. Check Sentry for rate limit errors (429 responses)
  2. Identify source IPs from logs
  3. If legitimate traffic → temporarily increase rate limits in middleware/security.ts
  4. If malicious → block IPs via Cloudflare WAF (if configured)
  5. Monitor rate_limit_counters table for patterns

Runbook: Security Breach

  1. Contain: Disable public access if possible (Railway service pause)
  2. Rotate immediately: JWT_SECRET, DB password, all API keys
  3. Audit: Query audit_trail and login_attempts tables for suspicious activity
  4. Assess scope: Which data was accessed? Which users affected?
  5. GDPR: If personal data compromised → notify DPA within 72 hours
  6. Fix: Patch vulnerability, deploy, verify
  7. Communicate: Notify affected users
